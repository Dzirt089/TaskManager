# TaskManager

Небольшой backend‑сервис для работы с задачами и справочником типов задач.

Проект сделан как **слоистый монолит** на ASP.NET Core: API принимает HTTP‑запросы, слой Application выполняет use case‑ы через MediatR, Domain хранит доменные сущности и простые инварианты, Infrastructure отвечает за EF Core и PostgreSQL.

Сервис умеет:
- создавать, читать, обновлять и удалять задачи;
- хранить и редактировать типы задач;
- фильтровать задачи по статусу и типу;
- сортировать и пагинировать список задач;
- возвращать ошибки в едином формате `ProblemDetails`.

## Стек

- **.NET 10 / ASP.NET Core Web API**
- **Entity Framework Core 10**
- **PostgreSQL 16**
- **MediatR** для разделения команд и запросов
- **FluentValidation** для валидации входных данных
- **Swagger / OpenAPI** для просмотра и ручного тестирования API
- **Docker Compose** для локального запуска PostgreSQL и, при желании, всего стека
- **MSTest** для unit‑тестов домена

## Быстрый старт

Есть два удобных сценария запуска:

1. **API локально + PostgreSQL в Docker** — основной вариант для разработки и отладки.
2. **Всё в Docker** — удобно, когда не хочется запускать API из IDE.

---

## Вариант 1. Запуск для разработки: API локально, PostgreSQL в Docker

### Что нужно заранее

- установлен **.NET 10 SDK**;
- установлен **Docker Desktop** или совместимый Docker Engine;
- свободны порты:
  - `5000` — API при локальном запуске;
  - `5432` — PostgreSQL.

### Шаг 1. Поднять PostgreSQL

Из корня репозитория:

```bash
docker compose up -d postgres
```

После старта база будет доступна по строке подключения:

```text
Host=localhost;Port=5432;Database=taskmanager;Username=postgres;Password=postgres
```

Она уже совпадает с `appsettings.json`, поэтому отдельно ничего менять не нужно.

### Шаг 2. Запустить API

Через CLI:

```bash
dotnet run --project src/TaskManager.Api/TaskManager.Api.csproj
```

Или через Visual Studio / Rider:
- открыть решение;
- выбрать стартовым проектом **TaskManager.Api**;
- нажать **Run / Debug**.

### Что произойдёт на старте

При запуске приложение:
- подключится к PostgreSQL;
- применит EF Core миграции;
- заполнит базу начальными типами задач (`Bug`, `Feature`, `Improvement`).

Это сделано автоматически в `Program.cs`, чтобы проект можно было быстро поднять локально без ручных SQL‑шагов.

### Куда заходить после запуска

- Swagger UI: `http://localhost:5000/swagger`
- Health check: `http://localhost:5000/health`

## Основные endpoints

### Типы задач

- `GET /api/task-types` — получить все типы задач
- `GET /api/task-types/{id}` — получить тип задачи по id
- `POST /api/task-types` — создать тип задачи
- `PUT /api/task-types/{id}` — обновить тип задачи
- `DELETE /api/task-types/{id}` — удалить тип задачи, если он не используется

Пример создания:

```http
POST /api/task-types
Content-Type: application/json

{
  "name": "Documentation"
}
```

### Задачи

- `GET /api/tasks` — список задач с фильтрацией, сортировкой и пагинацией
- `GET /api/tasks/{id}` — задача по id
- `POST /api/tasks` — создать задачу
- `PUT /api/tasks/{id}` — обновить задачу
- `DELETE /api/tasks/{id}` — удалить задачу

Пример создания задачи:

```http
POST /api/tasks
Content-Type: application/json

{
  "title": "Prepare release notes",
  "description": "Collect changes and publish release notes",
  "taskTypeId": "00000000-0000-0000-0000-000000000000",
  "dueDateUtc": "2026-05-01T12:00:00Z"
}
```

### Пример выборки с query‑параметрами

```http
GET /api/tasks?page=1&pageSize=20&status=InProgress&sortBy=CreatedAtDesc
```

## Как устроена обработка ошибок

Проект использует глобальный `GlobalExceptionHandler`.

Если внутри use case-а или валидации возникает ошибка, клиент получает **единый JSON‑ответ** в формате `ProblemDetails`:
- `400 Bad Request` — ошибка валидации или некорректный запрос;
- `404 Not Found` — сущность не найдена;
- `409 Conflict` — конфликт данных, например дубликат типа задачи;
- `500 Internal Server Error` — непредвиденная ошибка.

Это избавляет контроллеры от `try/catch` и делает ответы API предсказуемыми.

## Принятые архитектурные решения
### 1. Слоистый монолит вместо микросервисов

Для такого домена микросервисы были бы лишней сложностью. Здесь один bounded context, одна база и простой набор use case‑ов, поэтому выбран **слоистый монолит**:
- проще запустить локально;
- проще дебажить;
- меньше инфраструктурной нагрузки;
- нет сетевых вызовов между сервисами ради простого CRUD.

Это осознанный выбор в пользу **KISS**.

### 2. Разделение на Api / Application / Domain / Infrastructure

Слои разведены по ответственности:
- **Api** знает про HTTP, Swagger и форматы запросов/ответов;
- **Application** знает про сценарии использования системы;
- **Domain** знает про сущности и базовые бизнес‑правила;
- **Infrastructure** знает, как всё хранить в PostgreSQL.

Такой разрез удобнее поддерживать и расширять, чем один общий проект со всем подряд.

### 3. MediatR для use case-ов

Команды и запросы оформлены отдельно:
- `CreateTaskCommand`
- `UpdateTaskCommand`
- `GetTasksQuery`
- и т.д.

Почему так:
- контроллеры остаются тонкими;
- логика не размазывается по HTTP‑слою;
- проще тестировать use case отдельно;
- легче добавлять новые сценарии без разрастания сервисов‑комбайнов.

Это не “полноценный CQRS с разными хранилищами”, а **CQRS‑lite на уровне приложения**.

### 4. FluentValidation + pipeline behavior

Валидация вынесена из контроллеров и handler‑ов в validators.

`ValidationBehavior` запускает все подходящие валидаторы **до** выполнения handler-а. Это даёт:
- меньше дублирования;
- чище handlers;
- единый путь обработки ошибок валидации.

Такой подход хорошо ложится на MediatR pipeline.

### 5. EF Core без лишнего Repository Pattern

В проекте нет дополнительного слоя generic‑репозиториев поверх EF Core.
Вместо этого `Application` работает через `IApplicationDbContext`.

Почему это нормально здесь:
- сервис небольшой;
- EF Core уже даёт Unit of Work и change tracking;
- дополнительный generic repository часто только добавляет шаблонный код без пользы.

Это прагматичное решение в духе **YAGNI**.

### 6. PostgreSQL как основная БД

Выбран PostgreSQL, потому что это надёжная реляционная БД, хорошо подходящая для CRUD‑сервиса и локального запуска в Docker.

Плюсы для этого проекта:
- простое развёртывание;
- нормальная поддержка EF Core через Npgsql;
- предсказуемая работа с индексами, ограничениями и внешними ключами.

### 7. Fluent API-конфигурации отдельно от `Program.cs`

Конфигурации таблиц лежат в `TaskItemConfiguration` и `TaskTypeConfiguration`, а не в `Program.cs`.

Почему это сделано так:
- `Program.cs` должен собирать приложение, а не описывать схему БД;
- `DbContext` и конфигурации проще поддерживать отдельно;
- каждая сущность имеет свой собственный mapping-файл.

Это улучшает читаемость и снижает связность.

### 8. Автоматические миграции и seed на старте

При запуске приложение само:
- применяет миграции;
- добавляет базовые типы задач.

Для production‑систем это не всегда лучший вариант, но для небольшого сервиса и тестового задания это разумный компромисс:
- проект проще поднять;
- меньше ручных шагов;
- разработчику проще быстро получить рабочее состояние.

### 9. Глобальная обработка ошибок

Исключения не ловятся в каждом контроллере вручную. Вместо этого используется `GlobalExceptionHandler`, который переводит ошибки в `ProblemDetails`.

Плюсы:
- единый формат ошибок;
- меньше повторяющегося кода;
- корректные HTTP‑статусы;
- проще поддерживать клиентский контракт API.

### 10. Простая доменная модель без лишней “энтерпрайзности”

`TaskItem` и `TaskType` содержат только нужные инварианты: обязательность названия, trim строк, контроль корректных идентификаторов и т.д.

Проект сознательно не уходит в:
- event sourcing;
- сложные доменные сервисы;
- outbox/inbox;
- отдельные read/write базы.

Для текущего масштаба это был бы оверинжиниринг.

## Компромиссы и ограничения

Важно честно понимать, что проект специально оставлен простым.

Сейчас в нём нет:
- авторизации и аутентификации;
- optimistic concurrency;
- фоновых задач и очередей;
- отдельного слоя integration tests для PostgreSQL;
- сложного аудита изменений.

Это не недостаток само по себе — просто проект решает узкую задачу и не тянет лишнюю сложность заранее.
